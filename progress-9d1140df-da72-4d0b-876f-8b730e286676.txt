## Codebase Patterns
- Project uses Next.js 14 with App Router, TypeScript, Tailwind, shadcn/ui
- Tests use `node:test` + `tsx --test __tests__/**/*.test.ts`
- Test files use `__dirname` (not `import.meta.dirname`) because tsx runs as CJS
- Path alias `@/*` maps to project root (tsconfig paths)
- Prisma 5.x with SQLite, db file at `.data/financial.db`
- Singleton PrismaClient in `lib/db.ts` (globalThis pattern for dev hot-reload)
- Scripts: `npm run migrate` (prisma migrate dev), `npm run db:push` (prisma db push), `npm run db:seed` (tsx prisma/seed.ts)
- `.gitignore` excludes `.data/*.db`, `node_modules/`, `.next/`, `.env*`
- Server actions go in `lib/actions/` with `'use server'` directive at file top
- ActionResult<T> pattern: `{ success: true, data: T } | { success: false, error: string }` for typed error handling
- Seed data uses deterministic IDs prefixed with `seed-` for easy identification
- Server action tests work without Next.js runtime — 'use server' is a no-op when imported directly
- Test cleanup: use `deleteMany` with prefix filter on IDs (e.g., `txn-test-*`) for isolation
- Pure calculation functions in `lib/calculations.ts` — no DB deps, easy to test separately
- Holding updates are always done through `createTransaction` (atomic via `prisma.$transaction`)
- `getHoldings` accepts optional `currentPrices` map for computed fields (currentValue, gainLoss)
- Symbol normalization: `.trim().toUpperCase()` in transaction creation
- Pure utility modules (like csv-parser.ts, finnhub.ts, risk.ts) go in `lib/` directly, not in `lib/actions/`
- CSV parser uses simple split-based parsing with quoted field support — no external deps
- Import actions reuse existing server actions (e.g., `createTransaction`) for consistency rather than raw Prisma calls
- API client types go in `lib/types/` — separate type files per external API
- FinnhubClient uses mock mode when API key is unset — deterministic data seeded from symbol chars
- Rate limiting: sliding-window approach with `RateLimiter` class (reusable for other APIs)
- Static helper methods on client class (mapQuote, mapNewsItem, mockQuote, etc.) for easy testing
- Risk calculations use HoldingInput interface (symbol, quantity, avgCostBasis, optional currentPrice) — decoupled from Prisma model
- Concentration thresholds: >40% = HIGH, >25% = MEDIUM, otherwise LOW (strict greater-than, not >=)
- Diversification score uses weighted formula: 40% position count (asymptotic) + 60% Shannon entropy evenness
- Dashboard uses App Router nested layouts: app/dashboard/layout.tsx wraps all tab pages
- Dashboard layout is a client component ("use client") because it uses `usePathname` for active tab highlighting
- Tab pages are server components by default — only the layout needs client-side interop
- shadcn/ui components live in `components/ui/` — installed via `npx shadcn@latest add <component>`
- Dashboard navigation uses Link + pathname matching (not Tabs component) for proper URL-based routing
- Active tab detection: `pathname?.startsWith(tab.href)` — works for nested routes under each tab
- Tab page tests use filesystem checks (fs.existsSync + readFileSync) — no React rendering needed for structure validation
- Feature components go in `components/<feature>/` (e.g., `components/portfolio/`) — separate from `components/ui/`
- Per-holding concentration risk uses same thresholds as portfolio-level risk (>40 HIGH, >25 MEDIUM)
- Reusable `getRiskBadgeColor(risk)` helper maps LOW→green, MEDIUM→yellow, HIGH→red (defined in each component)
- Portfolio page is async server component — fetches data directly, no client-side state needed
- `formatCurrency` helper with Intl.NumberFormat for consistent EUR formatting across components
- Client components ("use client") for interactive forms; server components for data fetching pages
- Pattern: PortfolioActions wrapper embeds form + CSV dialogs, calls router.refresh() on success to re-render server component
- Dialog pattern: controlled open state + DialogTrigger for buttons, DialogContent for forms
- Form validation: export pure `validateXxxForm` function for testability, use it in the component's handleSubmit
- shadcn/ui Select, Dialog, Label components installed for form UI
- Signal engine uses strategy pattern: `SignalStrategy` function type, `DEFAULT_STRATEGIES` array, highest-confidence wins
- Signal actions use same ActionResult<T> pattern and symbol normalization as other actions
- Test isolation for signals: prefix `source` field with `sig-test-` and deleteMany on that prefix
- Server→Client data: serialize Prisma Date fields to ISO strings when passing from server components to client components (Date objects aren't serializable across the boundary)
- Decision queue duplicate guard: check for existing PENDING item before creating new DecisionQueueItem for a signal
- State machine pattern for workflow status: define VALID_TRANSITIONS map, check before any status update
- Decision actions use `prisma.$transaction` for atomic multi-model updates (e.g., executeDecision creates ExecutionLog + updates status)
- Test isolation for decisions: use test signal prefix (`dec-test-`) on source field, cascade delete from signals → decisions → execution logs
- Decision card conditional buttons: derive boolean flags (isPending, isApproved) from status, render action sections with `{flag && (...)}` — REJECTED/EXECUTED have no flags so no buttons
- Decision list dialog pattern: lift dialog state (open, action, decisionId, symbol) to list parent, pass confirm callback to dialog, call server actions in parent
- Status filter tabs: simple button list with `role="tablist"`, activeTab state, `filterByStatus` pure function for testability
